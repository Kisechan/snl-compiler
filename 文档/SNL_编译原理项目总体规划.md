# SNL 编译原理项目总体规划

## 1. 项目目标

本项目目标是设计并实现一个 SNL 语言编译器，完成编译原理课程实验要求，并按优秀档目标推进。

编译器输入为 SNL 源程序，输出包括：

- 词法分析结果：Token 序列。
- 语法分析结果：递归下降语法分析错误信息和语法树。
- 语义分析结果：语义错误信息。
- 目标代码生成结果：可在 MARS 等模拟器中运行的 32 位 MIPS 汇编代码。

最终演示时需要覆盖：

- 正确代码分析情况。
- 词法错误代码分析情况。
- 语法错误代码分析情况。
- 语义错误代码分析情况。
- 生成的 MIPS 目标代码在模拟器或虚拟机上的运行情况。

## 2. 资料依据与实现口径

本项目资料主要来自：

- 课程 PPT《编译原理实验》。
- PDF《编译程序的设计与实现》及其提取出的 DOCX/Markdown 资料。

实现口径：

- PPT 作为课程任务要求和 SNL 文法的主要依据。
- PDF 提取资料作为 Token、AST、符号表、语义分析和实现细节的辅助依据。
- 如果 PPT 与 PDF 提取资料冲突，默认以 PPT 为准。

已经确定的关键取舍：

- 使用 C++17 实现。
- 使用递归下降方法实现语法分析。
- 不实现 LL(1) 表驱动语法分析，除非后续重新要求。
- `return` 按 PPT 规则处理，即解析为 `return(expr)`。
- 目标代码选择 32 位 MIPS 汇编，而不是 PDF 中的 TM 虚拟机指令。
- Token 中 `LPAREN` / `RPAREN` 表示 `(` / `)`，`LMIDPAREN` / `RMIDPAREN` 表示 `[` / `]`。
- 类型系统中包含 `Char`，即使 PDF 中部分枚举提取结果没有列出该项。

## 3. 总体架构

编译流程设计为：

```text
SNL 源程序
  -> 词法分析
  -> Token 序列
  -> 递归下降语法分析
  -> AST 语法树
  -> 语义分析
  -> 带类型和符号绑定信息的 AST
  -> 中间表示或直接代码生成
  -> MIPS 汇编代码
```

建议项目采用如下模块划分：

- `token`：Token 类型、词素、语义值、源码位置。
- `lexer`：词法扫描器，处理保留字、标识符、常量、注释、分界符和词法错误。
- `ast`：语法树节点结构、节点类型、语法树打印。
- `parser`：递归下降语法分析器。
- `diagnostic`：统一错误报告，包含错误阶段、行列号和错误说明。
- `type`：语义类型系统，包括 integer、char、boolean、array、record、alias、procedure、error。
- `symbol_table`：分层符号表、作用域管理、标识符查找。
- `semantic`：声明处理、类型检查、过程调用检查、表达式类型标注。
- `ir`：可选中间代码层，用于降低 MIPS 生成复杂度。
- `mips`：32 位 MIPS 汇编代码生成。
- `main`：命令行入口。

## 4. 词法分析规划

词法分析器负责从 SNL 源程序中识别 Token。

需要支持的 Token 包括：

- 保留字：`program`、`procedure`、`type`、`var`、`if`、`then`、`else`、`fi`、`while`、`do`、`endwh`、`begin`、`end`、`read`、`write`、`array`、`of`、`record`、`return`、`integer`、`char`。
- 标识符：以字母开头，后接字母或数字。
- 整数常量：无符号整数。
- 字符常量：形如 `'a'`。
- 注释：`{ ... }`，注释内容不生成 Token。
- 运算符和分界符：`:=`、`=`、`<`、`+`、`-`、`*`、`/`、`(`、`)`、`[`、`]`、`.`、`;`、`,`、`..`。

需要检测的词法错误包括：

- 非法字符。
- 未闭合注释。
- 未闭合或格式非法的字符常量。
- 单独出现且不合法的符号组合。

词法分析输出建议包含：

```text
行号 | Token 类型 | 语义信息
```

## 5. 语法分析规划

语法分析只实现递归下降方法。

实现方式：

- 按 PPT 中 SNL 文法为每个非终结符编写一个解析函数。
- 语法分析输入为 Token 序列。
- 语法分析输出为语法错误信息和 AST 语法树。
- 第一版先保证正确程序能够稳定构造语法树。
- 第二版增加错误定位和适度错误恢复。

重点语法结构包括：

- 程序头、声明部分、程序体。
- 类型声明、变量声明、过程声明。
- 数组类型、记录类型、类型别名。
- 赋值语句、条件语句、循环语句、读写语句、过程调用语句、返回语句。
- 表达式、关系表达式、数组成员访问、记录字段访问。

语法树输出采用层次缩进文本形式，例如：

```text
ProK
  PheadK p
  TypeK
  VarK
  ProcDecK q
  StmLK
```

## 6. 语义分析规划

语义分析以 AST 为输入，建立符号表和类型信息，并检查语义错误。

符号表需要支持：

- 分层作用域。
- 当前作用域重复定义检查。
- 沿外层作用域链查找标识符。
- 类型名、变量名、过程名、参数、记录字段等不同类别。
- level、offset、类型信息、参数传递方式等属性。

类型系统需要支持：

- integer。
- char。
- boolean。
- array。
- record。
- alias。
- procedure。
- error。

需要检查的语义错误包括：

- 标识符重复定义。
- 使用未声明标识符。
- 标识符类别不符合预期。
- 类型别名引用无效。
- 数组下标类型错误或常量下标越界。
- 非数组变量使用 `[]`。
- 非记录变量使用 `.`。
- 记录字段不存在。
- 赋值语句左端不是可赋值变量。
- 赋值语句左右类型不兼容。
- 表达式运算分量类型不兼容。
- if / while 条件不是 boolean。
- 过程调用实参与形参数量不一致。
- 过程调用实参与形参类型不匹配。
- `var` 参数对应的实参不是可取地址变量。

## 7. MIPS 代码生成规划

目标代码生成输出 32 位 MIPS 汇编，优先保证能在 MARS 中运行。

基本约定：

- integer 使用 4 字节。
- char 为了简化栈帧和寻址，也按 4 字节存储。
- 数组元素连续存储。
- 记录字段按偏移顺序展开。
- `read` / `write` 使用 MARS syscall。

过程调用设计：

- 每次过程调用建立栈帧。
- 栈帧中保存返回地址、旧帧指针、静态链或 display 信息、参数、局部变量和临时值。
- 普通参数传值。
- `var` 参数传地址。
- 支持递归调用。
- 嵌套过程访问外层变量时，需要通过静态链或 display 定位外层活动记录。

代码生成顺序建议：

1. 先支持全局变量、integer 表达式、赋值、read、write。
2. 支持 if / while。
3. 支持数组和记录地址计算。
4. 支持普通过程调用。
5. 支持 `var` 参数。
6. 支持嵌套过程和递归。

## 8. 命令行接口规划

预期命令行接口如下：

```text
snlc --tokens input.snl
snlc --ast input.snl
snlc --semantic input.snl
snlc --mips input.snl -o output.asm
snlc --all input.snl
```

说明：

- `--tokens`：只运行词法分析并输出 Token 序列。
- `--ast`：运行词法分析和语法分析，输出语法树。
- `--semantic`：运行到语义分析阶段，输出语义错误信息。
- `--mips`：生成 MIPS 汇编代码。
- `--all`：依次输出词法、语法、语义和代码生成结果，便于最终演示。

## 9. 阶段安排

### 第一阶段：项目骨架与词法分析

- 建立 C++17 / CMake 项目结构。
- 定义 Token 类型和 Token 数据结构。
- 实现 Lexer。
- 输出 Token 序列。
- 准备词法正确和词法错误样例。

### 第二阶段：递归下降语法分析

- 定义 AST 节点结构。
- 按 PPT 文法实现递归下降解析函数。
- 构造语法树。
- 输出层次缩进语法树。
- 准备语法正确和语法错误样例。

### 第三阶段：语义分析

- 实现类型系统。
- 实现分层符号表。
- 遍历 AST 处理声明。
- 检查语句、表达式、变量访问和过程调用。
- 输出语义错误信息。

### 第四阶段：MIPS 基础代码生成

- 设计 MIPS 数据区、文本区和运行时约定。
- 支持 integer 变量、表达式、赋值、read、write。
- 支持 if / while。
- 在 MARS 中运行基础样例。

### 第五阶段：复杂结构与过程调用

- 支持数组地址计算。
- 支持记录字段地址计算。
- 支持过程调用、value 参数、var 参数。
- 支持嵌套过程和递归。

### 第六阶段：演示整理与回归测试

- 整理正确代码、词法错误、语法错误、语义错误、MIPS 运行样例。
- 固定输出格式。
- 编写项目说明。
- 准备最终演示流程。

## 10. 测试计划

测试样例按目录分类：

- `valid`：正确 SNL 程序。
- `lexical_errors`：词法错误程序。
- `syntax_errors`：语法错误程序。
- `semantic_errors`：语义错误程序。
- `mips`：用于验证 MIPS 运行结果的程序。

关键测试场景：

- 基本 program / begin / end 结构。
- integer 和 char 变量。
- 数组声明和访问。
- 记录声明和字段访问。
- if / while。
- read / write。
- 过程定义和调用。
- value 参数和 var 参数。
- 嵌套过程。
- 递归调用。
- 各类词法、语法、语义错误。

## 11. 风险与注意事项

- PPT 与 PDF 在 `return` 文法上存在冲突，本项目采用 PPT 的 `return(expr)`。
- PDF 中目标机是 TM，不是 MIPS，因此 MIPS 后端需要单独设计。
- 过程嵌套和非局部变量访问是代码生成阶段的主要难点。
- `var` 参数需要严格区分传值和传地址。
- 数组、记录、嵌套访问会影响语义分析和地址计算，需要尽早设计统一的左值表示。
- 最终演示需要稳定输出格式，因此 Token、AST、错误信息格式应尽早固定。
